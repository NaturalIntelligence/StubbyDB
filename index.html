Stubby DB3.0.0

Developed by a victim: Amit Gupta

What is it?

A framework to construct stubbed data services rapidly.

Why is it?

With usual approach

Set up

Initial code set up, logging, handling requests, server set up

Handling Response

generate dates, encode urls etc. Derive data from input

pieces of actual response
  • Code and stubs are tightly coupled.
  • Code for stub data
  • Not readable and difficult to understand

With Stubby DB

Features

Zero effort to setup & start

Single command for installation. 
$npm install stubby-db -g
Single command for start up and configuration 
$stubbydb
$stubbydb --help

Mappings

-  request:
      method: POST
      url: /stubs/(admin|staff|customer|security)/([0-9]+)
      post: password=([a-zA-Z0-9]+)
      headers:
            custom: (\w)

   response:
      status: 200
      latency: 3000
      body: >
        multiple line response
        another line
      #file: authentication.xml
      headers:
            custom: <% url.2 %>
  • Request part
  • Multiple mappings files
  • Short notations
  • Default mapping
  • Latency

Continue...

Strategy

-  request:

   response:
      strategy: first-found
      files: ["authentication.xml", "fault.xml"]
  • first-found
  • random
  • round-robin

Dumps

Dumps are static files. You can include dumps anywhere in response.

[[dumpspath:file1,file2]]
  • Consistency
  • Break the response into multiple parts

Markers

To make response dynamic

  • {{TODAY}}
  • {{TODAY+N}} , {{TODAY-N}}
  • {{TODAY-ny+nm-nd}}
  • {{URL:someurl}}

DB Sets

Skeleton based approach

-  request:
      url: /stubs/mix
      post: Hello ([a-zA-Z]+)!!
      method: POST

   dbset:
      db: Sample.txt
      key: '001'

   response:
      file: mix.txt
dbset 
    key |value      |markerval
    001 |1,2        |{{TODAY}}
    1   |value      |{{TODAY+1y-2m+3d}}

Debugging

  • On screen summary
  • debug.log, exceptions.log
  • ?debug=true
